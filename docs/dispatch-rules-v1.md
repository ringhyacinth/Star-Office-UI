# 牛马宗调度新规 v1

> 目标：在不牺牲质量的前提下，提升弟子利用率与并行度，缩短交付周期，降低会话超时与返工。

- 版本：v1.0
- 生效日期：即日
- 适用范围：task1-task16 全员
- 调度原则：**主岗优先、副岗补位、并行可控、质量前置、超时止损**

---

## 0. 总则（执行口径）

1. 所有任务统一由宗主（main）派发；跨部门协作通过主副岗矩阵执行。
2. 默认采用“**1主 + N辅**”并行模型，N 按任务分级自动确定。
3. 每个会话必须有：
   - 明确任务级别
   - 明确负责人（主岗）
   - 明确超时点与止损点
   - 明确质量门禁路径（开发→测试→审查）
4. 任何会话一旦达到止损条件，必须进入转交流程，不得无限续跑。

### 1.1 补位优先级规则

1. **同部门副岗优先**（沟通成本最低）。
2. 同部门饱和时，按“能力相邻”跨部门补位：
   - 开发区 ↔ 测试区（task1/2/16 与 task9/11）
   - 策划产品 ↔ 指挥中心（task4/8 与 task7/13）
   - 情报室 ↔ 金融区（task5/10/14/15 与 task6/12）
3. 同一弟子默认最多承接 **2 个活跃会话**（1个主会话 + 1个辅会话）。

## 2) 任务分级与默认并行人数规则

### 2.1 任务分级定义

| 级别 | 名称 | 典型场景 | 默认并行人数 | 建议组成 |
|---|---|---|---:|---|
| L0 | 快速响应 | 单点问答、轻量查询、文案微改 | 1 | 1主 |
| L1 | 标准执行 | 单模块开发/单次调研/常规修复 | 2 | 1主 + 1辅 |
| L2 | 跨域协作 | 开发+测试、方案+实现、需联调 | 3 | 1主 + 2辅 |
| L3 | 关键交付 | 多模块并行、上线前冲刺、里程碑交付 | 4-5 | 1主 + 3~4辅 |
| L4 | 紧急战情 | P0/P1事故、核心链路阻断、SLA风险 | 6-8 | 指挥+开发+测试+审查联合战情组 |

### 2.2 并行调度硬规则

1. 默认按级别并行，宗主可根据风险临时升降级。
2. **并行上限**：单任务会话最高 8 人，超过需拆分子任务树。
3. **WIP 限制**：
   - 单弟子主会话 ≤ 1
   - 单弟子总活跃会话 ≤ 2
4. 开发任务（L2及以上）必须至少绑定 1 个测试位（task9 或 task11）。
5. 关键任务（L3及以上）必须绑定审查位（task7），涉及安全时追加 task3。

### 2.3 自动升降级触发

- 升级条件（满足其一）：
  1) 阻塞 > 30 分钟且影响关键路径
  2) 返工达 2 轮
  3) 新增高风险项（安全/稳定/上线窗口）
- 降级条件（同时满足）：
  1) 风险关闭
  2) 剩余工作可由 ≤2 人完成
  3) 测试与审查排队可控

## 3) 会话止损 / 超时转交流程

### 3.1 超时阈值（按级别）

| 级别 | 首次检查点 | 止损阈值（未达阶段目标） | 强制转交阈值 |
|---|---:|---:|---:|
| L0 | 10 min | 20 min | 30 min |
| L1 | 30 min | 90 min | 120 min |
| L2 | 60 min | 180 min | 240 min |
| L3 | 120 min | 360 min | 480 min |
| L4 | 15 min | 30 min | 60 min |

### 3.2 止损触发条件

满足任一条件即触发止损：

1. 连续两次检查点无实质进展；
2. 阻塞依赖不可解且超过止损阈值；
3. 缺陷反复（同类问题第 3 次出现）；
4. 质量门禁连续失败 2 次。

### 3.3 转交流程（标准四步）

1. **冻结现场（5 分钟内）**：停止无效尝试，记录当前状态。
2. **交接打包（10 分钟内）**：提交“转交五件套”：
   - 任务目标与当前级别
   - 已完成内容与证据（文件/日志/测试结果）
   - 未完成项与明确阻塞点
   - 风险评估与建议接手人（主/副岗）
   - 下一步最小可行动作（Next 3 Steps）
3. **宗主裁决（10 分钟内）**：
   - 继续原执行人 + 增援
   - 更换主岗
   - 任务降级或拆分
4. **恢复执行**：新主岗接手后 15 分钟内回传新计划。

### 3.4 禁止事项

- 禁止无上限续跑（“再试一次”无限循环）。
- 禁止未交接直接弃坑。
- 禁止跳过宗主擅自改派。

## 4) 质量门禁（开发 → 测试 → 审查）

> 目标：并行提速但不牺牲质量。任何交付必须完整通过 3 道门。

### 4.1 门禁一：开发门（Dev Gate）

**责任位**：task1/task2/task16（按任务归属）

**准入标准（全部满足）**：
1. 代码可运行、无阻断错误；
2. 基础静态检查通过（lint/type/build）；
3. 关键路径具备最小自测证据；
4. 变更说明完整（改了什么、为什么改、风险点）。

### 4.2 门禁二：测试门（Test Gate）

**责任位**：task9（功能）+ task11（自动化/回归）

**准入标准（全部满足）**：
1. 冒烟测试通过；
2. 核心回归通过；
3. 缺陷分级达标：
   - P0 = 0
   - P1 = 0（或经宗主批准并有临时兜底）
   - P2 可带条件放行（必须有后续修复计划）
4. 测试报告可追溯（用例、结果、遗留问题）。

### 4.3 门禁三：审查门（Review Gate）

**责任位**：task7（质量审查）；高风险项增加 task3（安全审查）；需求一致性由 task4/task8 抽检。

**准入标准（全部满足）**：
1. 代码质量可维护（可读性、边界处理、回滚方案）；
2. 需求对齐（不偏题、不漏项）；
3. 安全/稳定风险可控；
4. 审查意见闭环完成。

### 4.4 质量门禁并行优化

1. 开发阶段即同步测试用例设计（前移测试）。
2. 测试阶段并行准备审查材料（缩短等待链路）。
3. 审查前置“风险预读”，减少一次性驳回率。

## 5) 每日运营指标与阈值告警

> 每日固定时间产出调度日报，指标统一按“绿/黄/红”三色管理。

### 5.1 核心指标定义

| 指标 | 定义 | 目标值（绿） | 预警（黄） | 告警（红） |
|---|---|---:|---:|---:|
| 弟子利用率 | 有效工作会话时长 / 可用时长 | ≥ 75% | 60%-74% | < 60% |
| 并行效率系数 | 实际并行任务数 / 计划并行任务数 | ≥ 0.85 | 0.70-0.84 | < 0.70 |
| 当日吞吐量 | 完成任务数（按级别加权） | ≥ 计划100% | 80%-99% | < 80% |
| 交付周期P90 | 任务从派发到通过审查的P90时长 | ≤ 基线110% | 111%-130% | > 130% |
| 返工率 | 进入二次以上返工任务占比 | ≤ 15% | 16%-25% | > 25% |
| 阻塞占比 | 阻塞时长 / 总时长 | ≤ 10% | 11%-20% | > 20% |
| 门禁一次通过率 | 一次性通过三道门任务占比 | ≥ 70% | 55%-69% | < 55% |
| 缺陷逃逸率 | 上线后发现缺陷 / 总缺陷 | ≤ 5% | 6%-10% | > 10% |
| 超时转交率 | 触发强制转交任务占比 | ≤ 8% | 9%-15% | > 15% |

### 5.2 告警处理动作

- **黄灯（预警）**：
  1) 当日内优化排班（补齐副岗）；
  2) 降低低优先任务并行；
  3) 针对性加测或加审。

- **红灯（告警）**：
  1) 立即召开战情调度（L4机制）；
  2) 冻结非关键新任务；
  3) 强制执行转交与复盘；
  4) 次日输出纠偏计划与负责人。

### 5.3 每日简报模板（建议）

1. 今日完成：L0/L1/L2/L3/L4 各多少
2. 门禁通过：一次通过率、主要驳回原因
3. 并行利用：利用率、并行效率、WIP 压力点
4. 风险列表：Top3 阻塞与处置进展
5. 明日排班：主岗安排、备用副岗、应急值守
