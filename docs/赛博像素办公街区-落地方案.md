# 赛博像素办公街区落地方案（star-office-ui）

> 大长老妙法天尊，妙计在此。目标是在现有 Flask + Phaser 架构上，快速落地“赛博像素办公街区”并支持后续扩展。

---

## 0. 目标与约束

### 目标
- 把当前“办公室大盘”升级为 **可交互的赛博像素街区**：可视化部门店铺、状态灯、电子看板、点击进入部门详情。
- 保持现有后端接口（`/live`、`/tasks`）兼容，优先增量改造。

### 约束
- 当前前端主要集中在 `frontend/index.html`（单文件，内联 CSS/JS）。
- 数据源来自 Flask：
  - Agent + 日志：`/live`
  - 任务：`/tasks`
- 需兼容桌面与移动端访问（现有 `Phaser.Scale.FIT` 已具备基础）。

## 1) 页面结构与组件拆分

### 1.1 方案对比

#### 方案A：继续单文件迭代（最低改造）
- 做法：继续在 `index.html` 增加街区逻辑。
- 优点：改动快，上线最快。
- 缺点：维护成本高，后续功能叠加易失控。

#### 方案B：原技术栈下模块化（推荐）
- 做法：保留 Flask + Phaser，不引入新框架；把内联脚本拆分为 ES Modules。
- 优点：风险低、可渐进上线、可测试性提升。
- 缺点：首轮需要一次性拆分成本。

#### 方案C：迁移到 React/Vue 组件体系（长期最优）
- 做法：前端重构，Phaser 作为子模块挂载。
- 优点：工程化最强，组件复用好。
- 缺点：工期大，不适合本次“快落地”。

**推荐结论：方案B。** 此方案妙在：不推翻现有可运行版本，1~2 周可交付可见成果。

## 2) Phaser 场景对象清单

### 2.1 场景分层
1. **BackgroundLayer**：街区地面、建筑底图
2. **ShopLayer**：六大部门店铺（情报/开发/测试/金融/策划/指挥）
3. **RoadLayer**：道路、导视标识
4. **AgentLayer**：角色精灵（现有 agent sprite 迁移）
5. **FXLayer**：霓虹、扫描线、粒子、雨滴（可选）
6. **UILayer**：状态灯、店铺铭牌、电子看板
7. **HitAreaLayer**：透明交互区（pointer 事件）

### 2.2 对象清单（建议）

| 类别 | 对象ID | 说明 | 数据来源 |
|---|---|---|---|
| 店铺 | `shop-info/dev/test/fin/plan/cmd` | 部门门店，点击进入详情 | 本地配置 + `/tasks` |
| 状态灯 | `lamp-{dept}` | 部门总体健康灯 | `/live` + `/tasks`聚合 |
| 电子看板 | `billboard-main` | 滚动展示日志/告警/任务 | `/live.logs` |
| 角色 | `agent-{id}` | 角色移动与状态表现 | `/live.state.agents` |
| 导航点 | `waypoint-*` | 角色寻路锚点 | 静态配置 |
| 交互区 | `hit-shop-*` | 店铺点击热区 | 静态配置 |
| 特效 | `neon-*` | 霓虹闪烁 | 本地动画 |

### 2.3 配置驱动（关键）
使用 `street.config.json` 管理坐标与行为，避免硬编码：

## 3) 电子看板日志滚动实现方案

### 3.1 数据流
1. 定时拉取 `GET /live`（3s）
2. 提取 `logs[agentId]` 最新条目
3. 生成唯一键去重：`key = agentId + ts + text`
4. 进入 `LogQueue`（队列长度建议 200）
5. `BillboardTicker` 每 1.2~1.8s 消费一条并滚动显示

### 3.2 渲染方案（推荐）

#### 方案1：Phaser 内实现（推荐）
- 组件：`Container + Graphics(mask) + BitmapText`
- 动画：`Tween`（Y 轴滚动）
- 优点：风格统一、性能稳定、与场景同步

#### 方案2：DOM/CSS 叠层
- 用绝对定位 DOM 覆盖 canvas 实现 marquee
- 优点：开发简单
- 缺点：移动端缩放时坐标同步麻烦

**推荐：方案1（Phaser 内实现）**

### 3.3 核心逻辑（伪代码）

```js
class BillboardTicker {
  queue = [];
  shown = new Set();

  ingest(logsByAgent) {
    for (const [aid, items] of Object.entries(logsByAgent)) {
      for (const item of items.slice(-2)) {
        const key = `${aid}|${item.ts}|${item.text}`;
        if (this.shown.has(key)) continue;
        this.shown.add(key);
        this.queue.push(formatLine(aid, item));
      }
    }
    if (this.queue.length > 200) this.queue.splice(0, this.queue.length - 200);
  }

  tick() {
    if (!this.queue.length) return;
    const line = this.queue.shift();
    this.renderAndScroll(line);
  }
}
```

### 3.4 样式规范
- 普通日志：`#8be9fd`
- 工具调用：`#f1fa8c`
- 错误/告警：`#ff5555`（带闪烁）
- 系统信息：`#bd93f9`

## 4) 状态灯与任务状态映射

### 4.1 映射原则
- **部门灯** 反映部门综合状态（Agent状态 + 任务状态）
- 优先级：`dead/error > doing > todo > done`

### 4.2 Agent 状态映射灯效

| Agent状态 | 灯色 | 动效 | 含义 |
|---|---|---|---|
| `dead` | `#4b4b4b` | 慢闪(2s) | 不可用 |
| `error` | `#ff3b30` | 快闪(0.6s) | 异常 |
| `executing/writing/researching/testing/reviewing/deploying/meeting` | `#00e676` | 呼吸(1.2s) | 忙碌运行 |
| `syncing` | `#b388ff` | 双脉冲 | 同步中 |
| `idle` | `#7f8c8d` | 常亮弱光 | 待命 |

### 4.3 任务状态映射灯效

| 任务状态 | 灯色 | 动效 | 权重 |
|---|---|---|---|
| `已完成` | `#4caf50` | 常亮 | 1 |
| `进行中` | `#2196f3` | 呼吸 | 3 |
| `待办` | `#ffc107` | 慢闪 | 2 |
| `已取消` | `#9e9e9e` | 常暗 | 0 |

### 4.4 部门聚合规则（建议）
- 若部门内存在 `dead/error` agent → 灯置红/灰优先。
- 否则若进行中任务占比 > 40% → 蓝绿呼吸。
- 否则若待办高于进行中 → 黄色慢闪。
- 否则绿色常亮。

## 5) 点击店铺跳转部门详情交互

### 5.1 交互链路
1. 用户点击店铺（`hit-shop-dev`）
2. Phaser 触发 `shop:click` 事件，携带 `deptId`
3. UI 打开右侧 `DeptDrawer`
4. 并行请求/计算：
   - 过滤本部门 Agent（`AGENT_HOME`）
   - 过滤本部门任务（`/tasks`）
   - 聚合最近日志（`/live.logs`）
5. URL 同步：`#dept=dev`（支持刷新回显）

### 5.2 详情面板内容
- 部门名 + 当前状态灯
- Agent 列表（状态、当前 detail、模型）
- 任务分组（待办/进行中/完成）
- 最近日志（最近 10 条）
- 操作按钮：`查看全部任务`（跳外部项目中心）

### 5.3 关键实现要点
- 点击热区用 `Phaser.Geom.Rectangle` + `setInteractive`
- 防误触：`pointerdown` + `pointerup` 均在热区才触发
- 移动端：抽屉改为底部弹层（`max-height: 70vh`）

## 风险与规避
- **风险1：单文件历史逻辑耦合高** → 先做“等价拆分”，再做功能增量。
- **风险2：日志量大导致滚动卡顿** → 队列上限 + 去重 + 节流渲染。
- **风险3：任务状态字段不规范** → 在 `mapping/statusMap.js` 做统一 normalize。
- **风险4：敏感配置管理风险** → 后端统一改为 `.env` 读取（建议优先处理）。
