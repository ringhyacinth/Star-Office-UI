# 办公室重构 v2 - 应急缺陷分诊看板（task15）

- 适用范围：`star-office-ui` 今晚上线窗口 + 上线后值班观察
- 目标：统一 P0/P1/P2 缺陷口径，降低高频消息下的沟通成本，保证可回归、可闭环
- 配套文档：
  - 验收口径：`docs/office-v2/task4-acceptance-spec.md`
  - 合并/回归清单：`docs/office-v2/task1-merge-checklist.md`

---

## 1. 分诊等级定义（P0 / P1 / P2）

| 等级 | 判定标准 | 典型示例 | 发布口径 |
|---|---|---|---|
| P0（阻塞） | 核心链路不可用、上线红线被击穿、存在持续性严重影响 | `/health` 非 200、首页白屏、`/live` 长时不可用、主流程不可操作 | 立即止血，必要时触发 NO-GO / 回滚 |
| P1（高优） | 核心功能降级明显但有临时绕过；影响可控但不可长期带病 | 日志筛选失效、任务区渲染错乱、关键指标延迟异常 | 当班内修复并回归，未闭环不得收工 |
| P2（中低优） | 不影响主链路，可延期处理；对体验或维护性有影响 | 样式错位、非关键动效缺失、文案/交互细节瑕疵 | 建单排期，纳入下个迭代 |

> 备注：若同一问题在值班窗口内影响范围扩大（单点 -> 面状），可从 P1/P2 升级到更高优先级。

## 3. 处理 SLA（响应与闭环时限）

### 3.1 时限矩阵

| 等级 | 首响（Ack） | 指派 Owner | 止血/绕过方案 | 修复或回滚决策 | 回归完成 | 状态播报频率 |
|---|---|---|---|---|---|---|
| P0 | ≤ 5 分钟 | ≤ 10 分钟 | ≤ 15 分钟 | ≤ 30 分钟 | ≤ 60 分钟（或进入回滚流程） | 每 10 分钟 |
| P1 | ≤ 10 分钟 | ≤ 20 分钟 | ≤ 60 分钟 | ≤ 4 小时 | 当班内完成（建议 ≤ 8 小时） | 每 30 分钟 |
| P2 | ≤ 4 小时 | ≤ 1 个工作日 | 可选 | ≤ 3 个工作日给出计划 | 按迭代排期 | 每日汇总 |

### 3.2 超时升级规则

1. 任一级别超过对应 SLA 未更新，自动升级提醒主负责人。
2. P0 在 30 分钟内无法稳定止血：直接进入回滚判定（参考 `task1-merge-checklist.md` F 章）。
3. P1 连续两次回归失败：升级为“P1-Blocker”，需增加 reviewer 并冻结相关改动面。

## 4. 回归入口（统一执行口径）

> 默认回归入口直接复用 `task1-merge-checklist.md` 的 D 章，避免多套标准。

### 4.1 通用入口

- D1 服务与接口冒烟：`/health` `/status` `/live` `/ollama`
- D2 前端主视图回归：首页加载、指标刷新、日志面板、任务仪表盘
- D3 同步链路回归：`start/check/stop task-sync` 三连

### 4.2 分级回归建议

| 等级 | 最低回归要求 | 通过标准 |
|---|---|---|
| P0 | D1 + D2（必要时 D3）全跑 | 核心链路恢复，连续观测无二次告警 |
| P1 | 相关模块专项 + D2 关键路径 | 功能恢复、无新增阻塞项 |
| P2 | 变更点专项回归 | 问题消失且不引入明显副作用 |

### 4.3 回归留痕最小要求

- 回归人
- 回归时间
- 回归步骤（引用 D1/D2/D3 编号即可）
- 结果（PASS/FAIL）
- 证据（命令返回/截图/日志）

---

## 5. 看板状态流转（建议）

`OPEN -> TRIAGED -> IN_PROGRESS -> FIXED -> REGRESSION_PASS -> CLOSED`

- 若回归失败：`FIXED -> REOPENED -> IN_PROGRESS`
- 若触发回滚：状态标记 `ROLLED_BACK`，并关联回滚记录

## 6. 高频消息下“最小回报模板”（主代理可直接复用）

> 目标：在消息风暴中 20~40 秒内给出一致、可执行的短回报。

```markdown
【分诊快报】
- 等级：P0/P1/P2
- 现象：
- 影响：
- Owner：
- 当前动作：止血中/修复中/回归中/已回滚
- 下一时间点：HH:mm 给出下一次更新
- 需要协助：无/需要 <资源或决策>
```

### 填写示例

```markdown
【分诊快报】
- 等级：P0
- 现象：首页白屏，`/health` 500
- 影响：主链路不可用
- Owner：task1
- 当前动作：止血中（回退到 stable tag）
- 下一时间点：19:20 更新
- 需要协助：需要 main 确认是否立即回滚
```

---

## 7. 值班执行建议（简版）

1. 先定级，再定 Owner，再发首条快报（不要反过来）。
2. 每次更新只讲 3 件事：影响变化、当前动作、下个时间点。
3. 修复完成不等于关闭，必须有回归结论与证据。
4. 文档记录与群内播报保持同一编号，减少信息错位。
