# 质量门禁 v1（多 Agent 协作）

> 适用范围：`star-office-ui`（Flask + 前端静态页面）
> 
> 目标：把“能跑”提升为“可交付、可回滚、可追责”。
> 
> 生效方式：**任一阻塞项不通过 = 不允许进入下一阶段**。

---

## 0. 角色与门禁规则

### 0.1 角色（最小配置）

- **变更负责人（Owner）**：提交代码的人（可由 Agent 执行）
- **审查负责人（Reviewer）**：至少 1 人，不能与 Owner 同人
- **联调负责人（Integrator）**：负责跨模块联调结论
- **发布负责人（Release）**：执行上线/回滚
- **值班负责人（Oncall）**：上线后 30~60 分钟观察

### 0.2 问题分级

- **P0（阻塞）**：安全、核心功能不可用、数据错乱/丢失、无法回滚
- **P1（高）**：主要流程可用但明显异常，有可接受临时绕过
- **P2（中低）**：体验/可维护性问题，不阻塞本次上线

### 0.3 通过标准

- 每个阶段都按以下格式留痕：
  - 检查项
  - 结果（PASS / FAIL）
  - 证据（日志、截图、命令输出、PR 评论链接）
  - 签字（Owner/Reviewer/Release）

---

## 1. PR 前检查（Gate-PR）

> 目标：在合并前把明显问题拦住。

## 必过项（阻塞）

- [ ] **变更说明完整**（目的、范围、影响面、回滚思路）
- [ ] **风险分级完成**（本次变更定级为 P0/P1/P2）
- [ ] **影响文件已自查**（尤其 `backend/app.py`、`frontend/index.html`、`scripts/*`）
- [ ] **无密钥/凭证泄漏**（代码、日志、截图、文档）
- [ ] **Python 语法检查通过**
  ```bash
  cd star-office-ui
  python3 -m py_compile backend/app.py backend/feishu.py backend/task_sync.py set_state.py
  ```
- [ ] **关键脚本可执行**
  ```bash
  cd star-office-ui
  bash scripts/check-task-sync.sh || true
  ```
- [ ] **核心接口冒烟通过**（至少本地）
  - `GET /health` 返回 200
  - `GET /status` 返回合法 JSON
- [ ] **文档同步更新**（README/TASK_SYNC_README/docs）
- [ ] **至少 1 位 Reviewer 通过**（含结论与风险备注）

## 建议项（不阻塞但强烈建议）

- [ ] 增加/更新最小测试用例（若当前仓库已有测试框架）
- [ ] PR 评论里附“手工验证步骤 + 结果截图”
- [ ] 对高风险变更附“5 分钟内回滚步骤”

---

## 2. 联调检查（Gate-INT）

> 目标：确认“多模块 + 多 Agent”协作链路真实可用。

## 必过项（阻塞）

- [ ] **联调环境一致**（Python 版本、`.env` 必填项、端口）
- [ ] **服务启动成功且稳定 10 分钟无崩溃**
  ```bash
  cd star-office-ui/backend
  python3 app.py
  ```
- [ ] **核心链路 1：任务大厅可打开，状态可刷新**
  - 打开前端页面成功
  - `/status` 更新后页面可感知
- [ ] **核心链路 2：任务同步脚本可启动/检查/停止**
  ```bash
  cd star-office-ui
  bash scripts/start-task-sync.sh
  bash scripts/check-task-sync.sh
  bash scripts/stop-task-sync.sh
  ```
- [ ] **失败恢复验证**
  - 人工停止同步器后可再次拉起
  - 后端重启后 `/health` 恢复 200
- [ ] **日志检查通过**
  - 无连续 traceback
  - 无明显重复报错刷屏

## 建议项（不阻塞）

- [ ] 模拟 2 个以上 Agent 并发更新状态，确认 UI 不错乱
- [ ] 对外网访问（如 tunnel）做一次可达性确认

---

## 3. 上线前检查（Gate-REL）

> 目标：上线动作可控，出问题能快速止损。

## 必过项（阻塞）

- [ ] **发布窗口确认**（谁发、何时发、谁值班）
- [ ] **发布版本冻结**（明确 commit/tag）
- [ ] **回滚版本已确认可用**（上一个稳定 tag/commit）
- [ ] **配置检查完成**（生产 `.env`、端口、权限）
- [ ] **上线步骤演练过一次**（至少在预发/本地完整演练）
- [ ] **上线后验证清单准备完成**
  - `/health` 200
  - `/status` 正常
  - 前端可访问
  - 同步器状态正常
- [ ] **监控与告警到位**（最小：日志可看、服务存活可查）

## 建议项（不阻塞）

- [ ] 分阶段发布（先小流量/先内部）
- [ ] 上线后 30 分钟内每 5 分钟一次巡检

---

## 4. 回滚标准（Gate-RBK）

> 目标：明确“何时必须回滚”，避免争论拖延。

## 4.1 触发条件（满足任一条，立即回滚）

- [ ] **P0 安全问题**（凭证泄漏、未授权访问等）
- [ ] **核心接口持续异常**：`/health` 或 `/status` 连续 2 分钟失败
- [ ] **核心功能不可用**：页面无法加载或关键交互不可用持续 5 分钟
- [ ] **同步链路中断**：任务同步无法恢复且影响业务可见性
- [ ] **错误率明显上升**：出现连续 traceback 且 5 分钟内无法止血

## 4.2 回滚执行步骤（标准化）

1. **宣布进入回滚**（在团队频道同步：原因、执行人、预计完成时间）
2. **停止当前发布动作**（停止新变更继续进入）
3. **切回上一个稳定版本**（tag/commit）
4. **重启服务并验证**（`/health`、`/status`、前端页面、同步脚本）
5. **确认恢复完成**（Reviewer + Release 双确认）
6. **记录复盘条目**（触发原因、发现时间、修复计划）

## 4.3 回滚完成判定

- [ ] 服务恢复可用并稳定 10 分钟
- [ ] 核心接口全部恢复
- [ ] 值班人员确认无新增 P0/P1 告警

---

## 5. 最小自动化建议（MVP，先做这 3 件）

## 建议 1：统一本地门禁脚本（最高性价比）

新增 `scripts/quality-gate-v1.sh`，串行执行：

1) Python 语法检查  
2) 基础密钥扫描（grep 常见 token 关键字）  
3) `/health` 冒烟检查（若本地服务已启动）

> 作为 PR 前“必跑脚本”，失败即不提 PR。

## 建议 2：最小 CI（PR 自动拦截）

新增 `.github/workflows/quality-gate-v1.yml`（如仓库启用 GitHub Actions）：

- 触发：`pull_request`
- 步骤：checkout → setup python → 执行 `scripts/quality-gate-v1.sh`
- 结果：失败则禁止合并

## 建议 3：回滚一键化（降低夜间故障成本）

新增 `scripts/rollback.sh <stable-tag-or-commit>`：

- 切版本
- 重启后端
- 检查 `/health`
- 输出 PASS/FAIL

> 目标：把“人工回滚 15~30 分钟”压缩到“5~10 分钟”。

---

## 6. 落地节奏（v1 → v1.1）

- **本周（v1）**：先执行本文手工清单 + `quality-gate-v1.sh`
- **下周（v1.1）**：接入 PR 自动化门禁（CI）
- **后续（v1.2）**：补充真实测试用例和发布仪表盘
